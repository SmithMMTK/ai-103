
ก่อนปรับใช้โมเดล คุณควรทำความเข้าใจว่าโมเดลทำงานได้ดีเพียงใดในหลายมิติ Benchmark ของโมเดลให้ข้อมูลเชิงวัตถุที่วัดผลได้ เพื่อช่วยให้คุณเปรียบเทียบโมเดลและตัดสินใจเลือกได้อย่างมีข้อมูลรองรับ Microsoft Foundry portal มีเครื่องมือ benchmark แบบครบถ้วน โดยจัดกลุ่มเป็นเมตริกด้านคุณภาพ ความปลอดภัย ต้นทุน และประสิทธิภาพ

## Access model benchmarks

คุณสามารถดู benchmark ภายใน Microsoft Foundry portal ได้ 2 วิธี:

ใน **model catalog** ให้ดู **Model leaderboard** เพื่อดูอันดับเปรียบเทียบของโมเดลทั้งหมดที่มีอยู่ มุมมองนี้ช่วยให้คุณระบุโมเดลที่โดดเด่นสำหรับเมตริกหรือสถานการณ์เฉพาะได้ โดย leaderboard จะแสดงโมเดลชั้นนำที่จัดอันดับตามคุณภาพ ความปลอดภัย ต้นทุนโดยประมาณ และ throughput

หากต้องการ benchmark แบบละเอียดของโมเดลใดโมเดลหนึ่ง ให้เปิด model card ของโมเดลนั้น แล้วเลือกแท็บ **Benchmarks** มุมมองนี้จะแสดงผลการทำงานของโมเดลในเมตริกและชุดข้อมูลต่างๆ พร้อมกราฟเปรียบเทียบกับโมเดลที่ใกล้เคียงกัน

## Quality benchmarks

Benchmark ด้านคุณภาพใช้ประเมินว่าโมเดลสร้างคำตอบได้แม่นยำ สอดคล้อง และเหมาะสมกับบริบทเพียงใด เมตริกเหล่านี้ใช้ชุดข้อมูลสาธารณะและวิธีประเมินมาตรฐาน เพื่อให้ผลลัพธ์มีความสม่ำเสมอ

**Quality index** ให้ภาพรวมระดับสูงโดยคำนวณค่าเฉลี่ยคะแนนความแม่นยำจากหลายชุดข้อมูล benchmark ซึ่งวัดด้านการให้เหตุผล ความรู้ การตอบคำถาม ความสามารถทางคณิตศาสตร์ และทักษะการเขียนโค้ด ค่า quality index ที่สูงกว่าหมายถึงประสิทธิภาพโดยรวมที่ดีกว่าในงานภาษาทั่วไป

Benchmark ด้านคุณภาพใช้ชุดข้อมูล เช่น:

- **Arena-Hard** - การตอบคำถามเชิงท้าทาย (adversarial)
- **BIG-Bench Hard** - ความสามารถด้านการให้เหตุผล
- **GPQA** - คำถามหลายสาขาระดับบัณฑิตศึกษา
- **HumanEval+** และ **MBPP+** - งานสร้างโค้ด
- **MATH** - การให้เหตุผลทางคณิตศาสตร์
- **MMLU-Pro** - การประเมินความรู้ทั่วไป
- **IFEval** - การทำตามคำสั่ง

คะแนน benchmark เป็นดัชนีที่ปรับให้อยู่ในช่วง 0 ถึง 1 โดยค่ายิ่งสูงยิ่งหมายถึงประสิทธิภาพที่ดีกว่า

## Safety benchmarks

เมตริกด้านความปลอดภัยช่วยให้มั่นใจว่าโมเดลจะไม่สร้างเนื้อหาที่เป็นอันตราย มีอคติ หรือไม่เหมาะสม Benchmark กลุ่มนี้สำคัญมากสำหรับแอปที่ผู้ใช้ปลายทางเข้าถึงได้ โดยเฉพาะในอุตสาหกรรมที่มีกฎกำกับหรือสถานการณ์ที่ต้องติดต่อกับลูกค้าโดยตรง

Microsoft Foundry ประเมินโมเดลในหลายมิติด้านความปลอดภัย:

**Harmful behavior detection** ใช้ benchmark ชื่อ HarmBench เพื่อวัดว่าโมเดลต้านทานการสร้างเนื้อหาไม่ปลอดภัยได้ดีแค่ไหน โดยการประเมินจะคำนวณค่า **Attack Success Rate (ASR)** ซึ่งค่ายิ่งต่ำยิ่งแปลว่าโมเดลปลอดภัยและมีความทนทานมากกว่า HarmBench ทดสอบ 3 ด้านหลัก:

- **Standard harmful behaviors** - อาชญากรรมไซเบอร์ กิจกรรมผิดกฎหมาย และอันตรายทั่วไป
- **Contextually harmful behaviors** - ข้อมูลบิดเบือน การคุกคาม และการกลั่นแกล้ง
- **Copyright violations** - การทำซ้ำเนื้อหาที่มีลิขสิทธิ์

**Toxic content detection** ใช้ชุดข้อมูล ToxiGen เพื่อวัดว่าโมเดลตรวจจับข้อความเกลียดชังแบบ adversarial และแบบแฝงได้ดีเพียงใด ค่า F1 ที่สูงกว่าหมายถึงประสิทธิภาพการตรวจจับที่ดีกว่า โดยเฉพาะในเนื้อหาที่กล่าวถึงกลุ่มชนกลุ่มน้อย

**Sensitive domain knowledge** ใช้ benchmark ชื่อ WMDP (Weapons of Mass Destruction Proxy) เพื่อวัดความรู้ของโมเดลด้านความปลอดภัยชีวภาพ ความปลอดภัยไซเบอร์ และความปลอดภัยทางเคมี ค่า WMDP ที่สูงกว่าหมายถึงโมเดลมีความรู้เกี่ยวกับความสามารถที่อาจเป็นอันตรายมากกว่า

คะแนนด้านความปลอดภัยช่วยให้คุณเข้าใจความทนทานของโมเดล ซึ่งสำคัญอย่างยิ่งสำหรับแอปที่ต้องใช้งานกับลูกค้าโดยตรง เพราะผลลัพธ์ที่เป็นอันตรายอาจก่อให้เกิดความเสี่ยงสูง

## Cost benchmarks

การเข้าใจผลกระทบทางการเงินจากการใช้งานโมเดล ช่วยให้คุณบาลานซ์ความต้องการด้านคุณภาพกับข้อจำกัดด้านงบประมาณ Benchmark ด้านต้นทุนใน Microsoft Foundry จะแสดงราคาสำหรับการปรับใช้แบบ serverless API และโมเดล Azure OpenAI

**Cost per input tokens** แสดงราคาสำหรับการประมวลผล input token จำนวน 1 ล้านโทเค็น (ข้อความที่คุณส่งเข้าโมเดล)

**Cost per output tokens** แสดงราคาสำหรับการสร้าง output token จำนวน 1 ล้านโทเค็น (ข้อความที่โมเดลสร้างออกมา)

**Estimated cost** รวมต้นทุนของ input และ output โดยใช้อัตราส่วนทั่วไป 3:1 (input 3 โทเค็นต่อ output 1 โทเค็น) เพื่อให้ได้ตัวเลขเดียวสำหรับเปรียบเทียบ ค่ายิ่งต่ำยิ่งคุ้มค่ากว่า

Benchmark ด้านต้นทุนช่วยให้คุณเลือกโมเดลที่ให้คุณภาพตามต้องการ ในราคาที่สอดคล้องกับรูปแบบการใช้งานและงบประมาณของแอปพลิเคชัน

## Performance benchmarks

เมตริกด้านประสิทธิภาพใช้วัดว่าโมเดลตอบสนองต่อคำขอได้เร็วและมีประสิทธิภาพเพียงใด Benchmark กลุ่มนี้สำคัญกับแอปแบบเรียลไทม์ที่ประสบการณ์ผู้ใช้ขึ้นอยู่กับความไวในการตอบสนอง

การวัด **Latency** ประกอบด้วย:

- **Latency mean** - เวลาเฉลี่ยเป็นวินาทีในการประมวลผลหนึ่งคำขอ
- **Latency P50** (median) - 50% ของคำขอเสร็จเร็วกว่าค่านี้
- **Latency P90** - 90% ของคำขอเสร็จเร็วกว่าค่านี้
- **Latency P95** - 95% ของคำขอเสร็จเร็วกว่าค่านี้
- **Latency P99** - 99% ของคำขอเสร็จเร็วกว่าค่านี้
- **Time to first token (TTFT)** - เวลาจน token แรกมาถึงเมื่อใช้การสตรีม

การวัด **Throughput** ประกอบด้วย:

- **Generated tokens per second (GTPS)** - จำนวน output token ที่สร้างได้ต่อวินาที
- **Total tokens per second (TTPS)** - จำนวน token รวม (input + output) ที่ประมวลผลได้ต่อวินาที
- **Time between tokens** - ช่วงเวลาระหว่างการได้รับ token ต่อเนื่องแต่ละตัว

Leaderboard สรุปประสิทธิภาพด้วยค่าเฉลี่ย time to first token (ยิ่งต่ำยิ่งดี) และค่าเฉลี่ย generated tokens per second (ยิ่งสูงยิ่งดี) โมเดลที่ throughput สูงและ latency ต่ำมักให้ประสบการณ์ที่ดีกว่าในแอปแบบโต้ตอบ สำหรับงาน batch processing ที่ความเร็วสำคัญน้อยกว่าต้นทุน คุณอาจให้น้ำหนักกับปัจจัยอื่นได้

## Use leaderboards and comparison features

Model leaderboard ช่วยให้คุณดูโมเดลชั้นนำตามเมตริกที่สนใจได้ คุณสามารถเรียงตามคุณภาพ ความปลอดภัย ต้นทุนโดยประมาณ และ throughput เพื่อหาโมเดลที่ตรงกับความต้องการมากที่สุด

**Scenario leaderboards** ช่วยให้คุณหาโมเดลที่ปรับเหมาะกับกรณีใช้งานเฉพาะ เช่น reasoning, coding, math, question answering หรือ groundedness หากแอปของคุณตรงกับสถานการณ์ใดชัดเจน ควรเริ่มจาก scenario leaderboard ที่เกี่ยวข้อง แทนการดูเฉพาะ quality index รวม

**Trade-off charts** แสดงเมตริกสองตัวพร้อมกัน เช่น quality เทียบ cost หรือ quality เทียบ throughput ภาพแสดงผลนี้ช่วยให้คุณหาจุดสมดุลที่เหมาะสมที่สุดกับความต้องการของคุณ ใช้เมนูดรอปดาวน์เพื่อเปรียบเทียบ quality กับ cost, throughput หรือ safety โมเดลที่อยู่ใกล้มุมขวาบนของกราฟมักทำได้ดีทั้งสองเมตริก บางครั้งโมเดลที่แม่นยำน้อยลงเล็กน้อย แต่เร็วขึ้นมากหรือถูกลงมาก อาจเหมาะกับงานของคุณมากกว่า

**Side-by-side comparison** ให้คุณเลือกโมเดล 2 หรือ 3 ตัวจาก leaderboard แล้วเปรียบเทียบกันหลายมิติ:

- Benchmark ด้านประสิทธิภาพ (quality, safety, throughput)
- รายละเอียดโมเดล (context window, ข้อมูลฝึก, ภาษาที่รองรับ)
- Endpoint ที่รองรับ (ตัวเลือกการปรับใช้)
- ความสามารถฟีเจอร์ (function calling, structured output, vision)
