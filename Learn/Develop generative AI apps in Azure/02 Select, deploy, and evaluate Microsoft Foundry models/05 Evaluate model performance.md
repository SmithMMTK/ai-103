การประเมินโมเดลที่คุณปรับใช้แล้วช่วยให้มั่นใจว่าเป็นไปตามมาตรฐานคุณภาพ ให้คำตอบได้อย่างถูกต้อง และพัฒนาได้อย่างต่อเนื่องในระยะยาว พอร์ทัล Microsoft Foundry มีแนวทางการประเมินหลายแบบ ตั้งแต่การทดสอบด้วยตนเอง ไปจนถึงเมตริกอัตโนมัติและโฟลว์การประเมินแบบครบวงจร

## Why evaluate models

การประเมินมีบทบาทสำคัญหลายด้านในการพัฒนาแอปพลิเคชัน Generative AI:

**Quality assurance** ช่วยระบุปัญหาและทำให้มั่นใจว่าโมเดลให้คำตอบที่ถูกต้องและตรงประเด็น การพบปัญหาในช่วง evaluation แทนที่จะไปพบใน production ช่วยปกป้องผู้ใช้และชื่อเสียงขององค์กร

**User satisfaction** จะดีขึ้นเมื่อโมเดลตอบได้อย่างสม่ำเสมอ มีประโยชน์ และเหมาะสม การประเมินช่วยให้คุณเข้าใจ user experience ของแอป และรู้ว่าควรปรับปรุงส่วนใดเพื่อให้เกิดผลสูงสุด

**Continuous improvement** เกิดจากการวิเคราะห์ผลประเมินเพื่อหาโอกาสพัฒนา การประเมินอย่างสม่ำเสมอเมื่อคุณปรับ prompt เพิ่ม feature หรือเทรนโมเดลใหม่ จะช่วยรักษาคุณภาพได้ต่อเนื่อง

**Compliance and safety** ช่วยยืนยันว่าโมเดลเป็นไปตาม policy หลีกเลี่ยงการสร้างเนื้อหาที่เป็นอันตราย และเคารพข้อกำหนดด้าน privacy และ data protection ของผู้ใช้

## Manual evaluation approaches

การทำ **Manual evaluation** คือการให้ผู้ประเมินที่เป็นมนุษย์ตรวจคำตอบของโมเดล แม้จะใช้เวลามาก แต่ให้ข้อมูลเชิงลึกที่ automated metrics จับได้ยาก

**Interactive testing** ใน playground ช่วยให้คุณสำรวจพฤติกรรมโมเดลในเชิงคุณภาพ คุณสามารถป้อน prompt ที่หลากหลาย สังเกตคำตอบ และจดปัญหา เช่น ข้อมูลผิด โทนภาษาไม่เหมาะสม หรือไม่ทำตาม instructions การทดสอบแบบสำรวจนี้ช่วยให้เข้าใจทั้งจุดแข็งและข้อจำกัดของโมเดล

เพื่อช่วยปรับแบบแอปพลิเคชันให้เหมาะสม คุณสามารถทดสอบโมเดลแบบเทียบกันใน playground โดยซิงก์ system instructions และพรอมป์เดียวกัน เพื่อเปรียบเทียบคำตอบได้โดยตรง

[![Screenshot of the chat playground in the Microsoft Foundry portal.](https://learn.microsoft.com/en-us/training/wwl-data-ai/model-catalog-evaluate/media/chat-playground.png)](https://learn.microsoft.com/en-us/training/wwl-data-ai/model-catalog-evaluate/media/chat-playground.png#lightbox)

**Structured review** คือการสร้างชุด test cases ที่แทน use cases ของแอป ผู้ประเมินจะให้คะแนนคำตอบตาม criteria เช่น:

- **Relevance**: คำตอบตรงกับคำถามหรือคำขอหรือไม่
- **Informativeness**: มีรายละเอียดเพียงพอและให้ข้อมูลที่เป็นประโยชน์หรือไม่
- **Engagement**: คำตอบน่าสนใจและมีโทนสนทนาที่เหมาะสมหรือไม่
- **Accuracy**: ข้อเท็จจริงและข้อความถูกต้องหรือไม่
- **Safety**: คำตอบหลีกเลี่ยงเนื้อหาที่เป็นอันตราย ลำเอียง หรือไม่เหมาะสมหรือไม่

โดยทั่วไปผู้ประเมินจะใช้ rating scale (เช่น 1-5) สำหรับแต่ละ criterion แล้วนำคะแนนจากหลาย test case มารวมกันเพื่อวัดคุณภาพโดยรวมเชิงปริมาณ

**User studies** คือการเก็บ feedback จากผู้ใช้จริงหรือผู้ใช้ตัวแทนที่ทดลองใช้งานแอป feedback เหล่านี้ช่วยเผยปัญหาในโลกจริงที่อาจไม่พบในการทดสอบแบบควบคุม เช่น ถ้อยคำที่ชวนสับสน บริบทไม่ครบ หรือความคาดหวังที่ไม่ได้รับการตอบสนอง

Manual evaluation ช่วยเสริมแนวทางอัตโนมัติ โดยเก็บมิติคุณภาพเชิงอัตวิสัย เช่น user satisfaction ความเหมาะสมตามบริบท และความสอดคล้องกับแบรนด์ ซึ่ง metrics เพียงอย่างเดียววัดได้ไม่ครบ

## Automated evaluation metrics

การประเมินอัตโนมัติใช้เมตริกมาตรฐานในการประเมินเอาต์พุตของโมเดลโดยอัตโนมัติ วิธีนี้ขยายสเกลได้ดีและให้ผลวัดที่สม่ำเสมอและเป็นกลาง

พอร์ทัล Microsoft Foundry รองรับเมตริกการประเมินหลายหมวดหมู่ ได้แก่:

**Generation quality metrics** ใช้วัดคุณภาพโดยรวมของคำตอบ:

- **Groundedness**: ตรวจว่าคำตอบอ้างอิงจากบริบทที่ให้มาหรือเป็นการคาดเดา Groundedness Pro ให้ผลแบบทวิภาค (grounded หรือไม่ grounded) ซึ่งเหมาะกับงานที่ต้องการความถูกต้องเชิงข้อเท็จจริง
- **Relevance**: วัดว่าคำตอบตอบโจทย์คำถามหรือคำขอของผู้ใช้อย่างเหมาะสมหรือไม่
- **Coherence**: ประเมินว่าคำตอบลำดับความคิดอย่างมีเหตุผลและคงความสอดคล้องของเนื้อหาหรือไม่
- **Fluency**: ประเมินความถูกต้องทางภาษาและความเป็นธรรมชาติของภาษา

**Risk and safety metrics** ใช้ตรวจหาเนื้อหาที่อาจเป็นอันตราย:

- **Self-harm content**: ตรวจพบคำตอบที่กล่าวถึงหรือส่งเสริมการทำร้ายตนเอง
- **Hateful and unfair content**: ระบุอคติ การเลือกปฏิบัติ หรือถ้อยคำแห่งความเกลียดชัง
- **Violent content**: แจ้งเตือนคำตอบที่มีหรือส่งเสริมความรุนแรง
- **Sexual content**: ตรวจพบเนื้อหาทางเพศที่ไม่เหมาะสม
- **Protected material**: ระบุความเสี่ยงในการทำซ้ำเนื้อหาที่มีลิขสิทธิ์หรือเป็นทรัพย์สินเฉพาะ
- **Indirect attack (jailbreak)**: ประเมินความเปราะบางต่อความพยายามบิดเบือนหรือหลบข้อจำกัด

สำหรับ content harm metrics ผลลัพธ์จะสรุปเป็น **defect rate** คือเปอร์เซ็นต์ของคำตอบที่มีระดับความรุนแรงเกิน threshold (โดยทั่วไปคือ Medium) ส่วน protected material และ indirect attack จะคำนวณ defect rate เป็น `(true instances / total instances) × 100`

เมื่อใช้การประเมินแบบมี AI ช่วย คุณต้องระบุโมเดล GPT ที่ทำหน้าที่ประเมิน โดย evaluator model นี้จะวิเคราะห์คำตอบของโมเดลที่คุณปรับใช้และให้คะแนนตาม criteria ที่เลือก

## Natural language processing metrics

NLP metrics เป็นการประเมินเชิงคณิตศาสตร์โดยไม่ต้องใช้ evaluator model โดย metrics เหล่านี้มักต้องใช้ข้อมูล ground truth หรือคำตอบที่คาดหวัง/ถูกต้องเพื่อเปรียบเทียบ

- **F1-score** วัดสัดส่วนคำที่ตรงกันระหว่างคำตอบที่สร้างกับคำตอบ ground truth โดยถ่วงดุลระหว่าง precision (หลีกเลี่ยงคำที่ผิด) และ recall (ครอบคลุมคำสำคัญ) จึงมีประโยชน์กับงานอย่างการจัดประเภทข้อความและการค้นคืนสารสนเทศ
- **BLEU** (Bilingual Evaluation Understudy) เปรียบเทียบ n-grams (ลำดับคำ) ระหว่างข้อความที่สร้างกับข้อความอ้างอิง นิยมใช้ประเมินงานแปลภาษา
- **METEOR** (Metric for Evaluation of Translation with Explicit Ordering) ต่อยอดจาก BLEU โดยคำนึงถึงคำพ้องความหมาย การตัดรากคำ และการถอดความ ทำให้เปรียบเทียบได้ยืดหยุ่นกว่า
- **ROUGE** (Recall-Oriented Understudy for Gisting Evaluation) ให้น้ำหนักกับ recall มากกว่า precision จึงเหมาะกับงานสรุปความที่การครอบคลุมประเด็นสำคัญสำคัญกว่าการหลีกเลี่ยงคำส่วนเกิน
- **GLEU** (Google-BLEU) เป็นตัวแปรของ BLEU ที่ออกแบบมาสำหรับการประเมินระดับประโยค

เมตริก NLP ทำงานได้ดีเมื่อมีคำตอบที่ถูกต้องชัดเจนหรือมีข้อความอ้างอิง แต่จะเหมาะน้อยลงกับงานสร้างคำตอบแบบปลายเปิดที่มีคำตอบถูกต้องได้หลายแบบ

## Create comprehensive evaluations

ฟีเจอร์ **Evaluation** ในพอร์ทัล Microsoft Foundry ช่วยให้คุณรันการประเมินอย่างเป็นระบบ โดยใช้ test dataset และหลาย metrics พร้อมกัน

คุณสามารถตั้งฐานการประเมินได้จากหนึ่งในตัวเลือกต่อไปนี้:

- **Model**: ประเมินโมเดลที่ปรับใช้แล้วด้วยพรอมป์ที่คุณกำหนด ระบบจะสร้างเอาต์พุตระหว่างการประเมิน
- **Agent**: ประเมินคำตอบของเอเจนต์ด้วยพรอมป์ที่ผู้ใช้กำหนด
- **Dataset**: ประเมินเอาต์พุตที่สร้างไว้ล่วงหน้าและมีอยู่แล้วในชุดข้อมูลทดสอบ

เมื่อประเมินโมเดลหรือเอเจนต์ คุณต้องมี dataset เพื่อป้อน input สำหรับการประเมิน โดยมี 3 ทางเลือก:

- **Upload new dataset**: อัปโหลดไฟล์ CSV หรือ JSONL ที่มีกรณีทดสอบจากเครื่องของคุณ
- **Use existing dataset**: เลือกจากชุดข้อมูลที่เคยอัปโหลดไว้ในโปรเจ็กต์
- **Generate synthetic dataset**: หากยังไม่มีข้อมูลทดสอบ ระบบสามารถสร้างข้อมูลตัวอย่างจากคำอธิบายหัวข้อที่คุณให้มาได้ คุณระบุทรัพยากรที่ใช้สร้างข้อมูล จำนวนแถว และพรอมป์ที่อธิบายข้อมูลที่ต้องการได้ รวมถึงอัปโหลดไฟล์เพื่อเพิ่มความเกี่ยวข้องกับงานเฉพาะของคุณ

สำหรับการประเมินแบบ dataset ที่มี output สร้างไว้แล้ว ให้เลือกหรืออัปโหลดชุดข้อมูลที่มีทั้ง input และคำตอบที่โมเดลสร้าง

หลังจากตั้งค่า metrics ที่ต้องการคำนวณ การแมปฟิลด์ของข้อมูลประเมิน และ system prompt ของโมเดลแล้ว คุณสามารถเริ่ม evaluation job ได้ ซึ่งอาจใช้เวลาเนื่องจากระบบจะประมวลผลแบบ asynchronous โดยไล่ประเมินทีละแถวใน test dataset ตาม metrics ที่เลือก

## Explore the evaluator library

**Evaluator library** เป็นศูนย์กลางสำหรับดูและจัดการ evaluator ที่มีทั้งหมด คุณเข้าถึงได้จากหน้า **Evaluation** ของโปรเจ็กต์ โดยเลือกแท็บ **Evaluator library**

ใน evaluator library คุณสามารถ:

- ดู evaluator ที่ Microsoft คัดสรรไว้สำหรับ quality, safety และ performance
- ตรวจรายละเอียด evaluator เช่น name, description, parameters และไฟล์ที่เกี่ยวข้อง
- ทบทวน annotation prompts ของ quality evaluators เพื่อเข้าใจวิธีคำนวณ metrics
- ตรวจ definitions และ severity levels ของ safety evaluators
- จัดการ custom evaluators ที่คุณสร้างสำหรับสถานการณ์เฉพาะ

ไลบรารีรองรับการจัดการเวอร์ชัน ทำให้คุณเปรียบเทียบเวอร์ชันต่างๆ กู้คืนเวอร์ชันก่อนหน้าเมื่อจำเป็น และทำงานร่วมกับผู้อื่นบน custom evaluators ได้

## Iterate based on evaluation

evaluation results ช่วยชี้แนวทางขั้นตอนถัดไป:

เมื่อคะแนนต่ำกว่าระดับที่ต้องการ ให้พิจารณา:

- **Prompt engineering**: ปรับแต่งคำสั่งและ system messages ให้ชัดเจนขึ้น
- **Different models**: ทดลองโมเดลอื่นที่เหมาะกับ use case ของคุณมากกว่า
- **RAG integration**: เพิ่มความสามารถ retrieval เพื่อยึดคำตอบกับข้อมูลของคุณ
- **Fine-tuning**: เทรนโมเดลเพิ่มเติมบนโดเมนเฉพาะของคุณ (ถ้ารองรับ)

แต่ละแนวทางอาจเพิ่ม complexity (และบางครั้งเพิ่ม cost) ดังนั้นควรคำนึงถึงเรื่องนี้ขณะวางแผนปรับปรุง

เมื่อ safety metrics บ่งชี้ความเสี่ยง:

- **Content filters**: ใช้บริการ Azure AI Content Safety
- **Prompt hardening**: เพิ่มคำสั่งด้านความปลอดภัยใน system messages
- **Output validation**: ตรวจสอบคำตอบก่อนแสดงให้ผู้ใช้

การประเมินเป็นประจำเมื่อมีการเปลี่ยนแปลงช่วยติดตามพัฒนาการและป้องกันคุณภาพถดถอย ควรกำหนด evaluation benchmarks ตั้งแต่ช่วงต้นของการพัฒนา แล้วประเมินซ้ำหลังปรับแก้เพื่อวัด impact อย่างเป็นกลาง

เมื่อผสานการทดสอบแบบแมนนวล เมตริกอัตโนมัติ และโฟลว์ประเมินแบบครบวงจรเข้าด้วยกัน คุณจะมั่นใจได้มากขึ้นว่าโมเดลทำงานได้ดี ให้บริการผู้ใช้อย่างปลอดภัย และตอบโจทย์มาตรฐานคุณภาพของแอปพลิเคชัน
